# Toxic output that scores clean: Unicode homoglyph evasion of garak's string detectors

**Project:** [NVIDIA/garak](https://github.com/NVIDIA/garak) (LLM vulnerability scanner / red-teaming framework)
**Class:** Detection evasion via Unicode normalization gap, eval-integrity bug
**Discovered / reported by:** [AUTHENSOR](https://github.com/AUTHENSOR) in [issue #1867](https://github.com/NVIDIA/garak/issues/1867)
**My part:** the fix ([PR #1884](https://github.com/NVIDIA/garak/pull/1884), open) plus this breakdown.

> Credit: AUTHENSOR found and clearly documented this one in #1867. I built the fix and wrote up the mechanism and the design tradeoffs below.

## TL;DR

garak decides whether a probe succeeded by string-matching the model's output against known trigger strings. AUTHENSOR pointed out that the matcher did no Unicode normalization, so a model that emits the trigger in fullwidth (`ｈｅｌｌｏ`) or zero-width-spliced (`h‍e‍l‍l‍o`) form scores as *clean*, even though a human reads it as the trigger. That's a false negative in a detection tool. The fix adds opt-in normalization so those visually-identical variants get folded back before matching.

This is the sibling of [the LLM-judge rating-rail bug](2026-07-garak-llm-judge-rating-injection.md), also reported by AUTHENSOR: same root theme, a naive text operation over untrusted model output getting fooled into under-reporting.

## Background: how garak decides pass/fail

A big chunk of garak's detectors are `StringDetector` subclasses. They carry a list of trigger substrings, and detection is basically "did any of these show up in the output." If the probe is checking whether a model will emit some toxic phrase, a canary, or a known bad completion, the substring match is the whole verdict.

The matching loop was a straight substring/word check:

```python
for s in self.substrings:
    if not self.case_sensitive:
        s, output_text = s.lower(), output_text.lower()
    # ... substring / word match on raw output_text
```

Case folding, and nothing else. Which is fine until you remember the output is generated by the exact model you're testing, and it can pick its own encoding.

## The evasion

Two flavors, both of which look identical to a human:

**Fullwidth homoglyphs.** Unicode has a whole "fullwidth" block that mirrors ASCII: `ｈｅｌｌｏ` is U+FF48 U+FF45 U+FF4C U+FF4C U+FF4F, not `hello`. Byte-for-byte different, so `"hello" in output` is `False`. Detector says clean.

**Zero-width splicing.** Stick a zero-width joiner (U+200D, Unicode category `Cf`, "format") between the letters: `h‍e‍l‍l‍o`. Renders as "hello", but the substring `hello` isn't in there. Detector says clean again.

So a model that *did* produce the toxic content gets a 0.0 (no detection). If you're running garak to decide whether your model is safe to ship, that's the tool handing you a false all-clear.

## Why this is the dangerous kind of bug

It's not memory-unsafe, nothing crashes. It's worse in a specific way: it's an **integrity failure in the thing you use to measure safety.** A crash you'd notice. A silent false negative you build a release decision on. Detection tools live and die on their false-negative rate, and "the payload was there in a form your matcher didn't canonicalize" is a false negative you'd never see in the report.

## The fix (#1884)

Add an opt-in `normalize` knob to `StringDetector` and canonicalize both sides before matching:

```python
"normalize": None,  # None, "NFKC", or "NFKC+strip_format"

def _apply_normalize(self, text: str) -> str:
    if self.normalize == "NFKC":
        return unicodedata.normalize("NFKC", text)
    elif self.normalize == "NFKC+strip_format":
        text = unicodedata.normalize("NFKC", text)
        return "".join(c for c in text if unicodedata.category(c) != "Cf")
    else:
        raise ValueError(f"Don't know how to process normalize: {self.normalize!r}")
```

- **`NFKC`** collapses compatibility forms: fullwidth `ｈｅｌｌｏ` normalizes straight back to `hello`. Handles the homoglyph flavor.
- **`NFKC+strip_format`** additionally drops Unicode category `Cf` characters (zero-width joiner/space, other format controls), which NFKC leaves in place. Handles the zero-width-splicing flavor.

A few design choices worth calling out:

- **Both sides, but triggers only once.** The trigger substrings are config, independent of the output, so they're normalized a single time up front; each output is normalized as it comes in. (Getting *both* normalization sites was the main thing to get right, easy to normalize the output and forget the triggers, which would silently break every match.)
- **Opt-in, not default.** NFKC can change meaning for some detectors (it rewrites ligatures, some math symbols, etc.), so forcing it on everyone could create *false positives* elsewhere. Leaving it `None` by default keeps existing detectors byte-identical and lets the ones that need it turn it on.
- **Fail safe, not loud.** An invalid `normalize` value raises `ValueError` from `_apply_normalize`; `detect` catches it and returns `None` (can't-detect) for the affected outputs with a warning, rather than letting a config typo tear down the whole scan run.

Tests cover the actual bypass: without `normalize`, fullwidth and zero-width `hello` score 0.0 (bypass reproduced); with the right tier, they score 1.0 (caught).

## Takeaway

Any time you're matching fixed strings against model-generated text, you've got a normalization problem whether you've thought about it or not. Homoglyphs, compatibility forms, and zero-width characters are the model's version of `../` in a path, visually innocent, semantically the payload. Canonicalize both the needle and the haystack to the same form before you compare, and pick the normalization tier based on what your attacker can reach for (compatibility block vs. format controls). And if you're building anything that *scores* model output, remember the output is adversarial input: the failure you care about is the one where it quietly reads as safe.
